# T02 — Счётчик ошибок ИНН + двухэтапная эскалация

## Customer-facing инкремент

После двух неудачных попыток ввести ИНН (невалидный формат или не найден в Bitrix) Алина сама предлагает написать в поддержку — клиент не застревает в бесконечной петле.

## Цель

Включить поле `error_count` в сессии: инкрементировать при каждой ошибке в `WAITING_INN`, на 2-й и последующих ошибках добавлять в AI-ответ упоминание `@Lobster_21`.

## Scope

- `handlers/text.py` (обработка `WAITING_INN`):
  - После неудачи (невалидный ИНН или не найден) увеличить `session["error_count"]` и сохранить в Supabase
  - Передавать `error_count` в вызовы `openai_svc.no_inn_in_text()` и `openai_svc.inn_not_found()`
- `services/openai_client.py`:
  - В промпты `no_inn_in_text` и `inn_not_found` добавить параметр `escalate: bool`
  - Если `escalate=True` → в конце ответа Алина упоминает `@Lobster_21`

### Правило счётчика

- `error_count` инкрементируется при любой ошибке ИНН (невалидный формат + не найден в Bitrix)
- При `error_count >= 2` → `escalate=True`
- Сбрасывается только при `/start`

## Out of scope

- Блокировка клиента после N попыток
- Отправка уведомлений менеджеру
- Счётчик для ошибок телефона (это T03)

## Как протестировать

1. Написать невалидный текст (без 12-значного ИНН) → бот отвечает без упоминания поддержки
2. Написать второй невалидный текст → в ответе есть `@Lobster_21`
3. Написать несуществующий ИНН (12 цифр, но не в Bitrix) → если это уже 2-я+ ошибка — в ответе `@Lobster_21`
4. Написать `/start` → счётчик сбрасывается (следующая ошибка снова без эскалации)
5. Проверить в Supabase: `error_count` обновляется корректно

## Критерии приёмки

1. 1-я ошибка ИНН → AI-ответ без `@Lobster_21`
2. 2-я+ ошибка ИНН → AI-ответ содержит `@Lobster_21`
3. Оба типа ошибок (невалидный формат и не найден) считаются одним счётчиком
4. `/start` сбрасывает `error_count` до 0
5. Поле `error_count` в Supabase обновляется при каждой ошибке
