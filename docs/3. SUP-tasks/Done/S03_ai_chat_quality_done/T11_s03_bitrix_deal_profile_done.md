# T11 — Обогащение контекста клиента через Bitrix

## Customer-facing инкремент

Клиент получает ответ, основанный на реальном статусе его дела: ИИ знает текущую стадию процедуры и какие документы уже сданы — и не выдаёт общий чеклист тому, кто давно всё предоставил.

## Цель

Исправить `get_deal_profile(deal_id)` в `BitrixService`: метод уже существует, но читает статус документов из пустых полей сделки (`UF_CRM_...`). Реальный чеклист документов хранится в задаче «Собрать ЛИЧНЫЕ документы клиента», привязанной к сделке через поле `UF_CRM_TASK = ["D_{deal_id}"]`. Нужно переключиться на чтение чеклиста из этой задачи.

## Что уже реализовано (не трогать)

- `STAGE_LABELS` в `services/bitrix.py` — маппинг стадий воронки БФЛ ✅
- Базовая структура `get_deal_profile()` — batch-запрос, форматирование, возврат `""` при ошибке ✅
- Параметр `deal_profile: str = ""` в `support_svc.answer()` ✅
- Вызов `get_deal_profile()` в `handlers/text.py` перед `answer()` ✅
- `deal_id` сохраняется в `bot_sessions` при авторизации ✅

## Что нужно изменить

### `services/bitrix.py`

**Удалить:**
- Константу `DOCUMENT_FIELDS` — поля сделки не используются для документов (они пустые)

**Изменить `get_deal_profile(deal_id)`:**

Вместо одного batch-запроса с полями сделки — **два последовательных batch-запроса**:

**Batch 1** (оба запроса нужен только `deal_id`, запускаются параллельно):
```
cmd[deal]  = crm.deal.get?ID={deal_id}&select[]=STAGE_ID&select[]=ASSIGNED_BY_ID
cmd[tasks] = tasks.task.list?filter[UF_CRM_TASK]=D_{deal_id}
                             &filter[%TITLE]=Собрать ЛИЧНЫЕ
                             &select[]=ID
                             &start=0
```

**Batch 2** (нужны результаты Batch 1):
```
cmd[manager]   = user.get?ID={ASSIGNED_BY_ID из deal}
cmd[checklist] = tasks.task.get?taskId={ID из tasks[0]}&select[]=CHECKLIST
```

**Формирование чеклиста из результата Batch 2:**
- Из `checklist` — взять только пункты с `parentId != 0` (пропустить корневой `BX_CHECKLIST_1`)
- `isComplete = "Y"` → `✅ {title}`
- `isComplete = "N"` → `⏳ {title}` (показывать только если есть хотя бы один ✅, иначе — «документы ещё не начали сдавать»)
- Обрезать title до 60 символов для краткости промпта

**Формат вывода:**
```
=== ПРОФИЛЬ КЛИЕНТА (Bitrix) ===
Стадия дела: Реализация имущества (без имущества)
Ответственный менеджер: Лебедева Алена Дмитриевна

Чеклист документов (задача Bitrix):
  ✅ Паспорт (все страницы без обложки) - ПРЕДОСТАВЛЯЕТ КЛИЕНТ
  ✅ СНИЛС - ПРЕДОСТАВЛЯЕТ КЛИЕНТ
  ⏳ Трудовая книжка (все страницы...)
  ⏳ Справка 2-НДФЛ за последние 3 года...
```

Если задача не найдена — раздел чеклиста не выводить (только стадия и менеджер).
При любой ошибке — вернуть `""`.

## Как протестировать

1. Авторизоваться с ИНН клиента, у которого в Bitrix есть задача «Собрать ЛИЧНЫЕ документы клиента» с частично заполненным чеклистом
2. Спросить: «Какие документы мне нужно ещё сдать?»
3. Проверить: ответ называет конкретные непредоставленные документы из чеклиста задачи
4. Спросить: «На какой стадии моё дело?»
5. Проверить: ответ называет реальную стадию, а не общую фразу

## Критерии приёмки

1. `DOCUMENT_FIELDS` удалена из `services/bitrix.py`
2. `get_deal_profile()` использует 2 batch-запроса: Batch1 (deal + tasks.task.list), Batch2 (user + tasks.task.get с CHECKLIST)
3. Чеклист строится из пунктов задачи (parentId != 0), а не из полей сделки
4. Если задача не найдена — метод всё равно возвращает стадию и менеджера
5. Если Bitrix недоступен — бот продолжает работу без профиля (graceful degradation)
6. Ответ ИИ отражает реальный статус документов из задачи

## Технические детали (из исследования API)

- Задача привязана к сделке: `ufCrmTask = ["D_{deal_id}"]`
- Фильтр поиска задачи: `filter[UF_CRM_TASK]=D_{deal_id}` + `filter[%TITLE]=Собрать ЛИЧНЫЕ`
- Чеклист получается через: `tasks.task.get?taskId=X&select[]=CHECKLIST`
- Структура чеклиста: `{id: {id, parentId, title, isComplete, ...}}`
- Корневой элемент: `parentId = 0`, title = `BX_CHECKLIST_1` — пропускать
- Реальные пункты: `parentId != 0`
