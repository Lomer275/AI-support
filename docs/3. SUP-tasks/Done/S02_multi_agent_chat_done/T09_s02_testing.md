# T09 — Тестирование multi-agent chat

## Customer-facing инкремент

Убедиться что все сценарии работают корректно перед релизом: клиент получает осмысленные ответы, история сохраняется, граничные случаи не роняют бота.

## Цель

Пройти полный набор тест-сценариев вручную через Telegram. Зафиксировать результат по каждому сценарию.

## Тест-сценарии

### 1. Типовой вопрос с документами
- Авторизоваться клиентом у которого есть записи в `smart_process_kad_arbitr`
- Написать: «Когда завершится моё дело?»
- Ожидаемо: осмысленный ответ со ссылкой на этап/даты из документов; «печатает...» всё время ожидания

### 2. Вопрос без документов
- Использовать ИНН без записей в `smart_process_kad_arbitr` (или временно очистить)
- Написать любой вопрос
- Ожидаемо: бот отвечает без ошибок, без `[Данные не найдены]` в тексте клиенту

### 3. Память диалога
- Написать вопрос 1: «Могут ли у меня забрать квартиру?»
- Написать вопрос 2: «А что насчёт машины?»
- Ожидаемо: ответ на вопрос 2 учитывает контекст вопроса 1 (Алина помнит про квартиру)

### 4. Персистентность истории
- Написать вопрос, получить ответ
- Перезапустить бота (`docker compose restart bot`)
- Написать следующий вопрос
- Ожидаемо: история сохранилась, Координатор видит предыдущий обмен

### 5. Fallback при падении pipeline
- Временно сломать `OPENAI_MODEL_SUPPORT` (невалидное значение)
- Написать вопрос
- Ожидаемо: клиент получает fallback-ответ Алины, не технический стектрейс; в логах — ошибка с traceback

### 6. switcher=true (заглушка)
- Написать: «Хочу записаться на консультацию к юристу»
- Ожидаемо: клиент получает осмысленный ответ (не падение); `switcher` не ломает ответ

### 7. Регрессия авторизации
- Пройти полный флоу: `/start` → ввод ИНН → поделиться телефоном → авторизация
- Написать вопрос в WAITING_INN (до авторизации)
- Ожидаемо: всё работает как в S01, старые промпты не затронуты
