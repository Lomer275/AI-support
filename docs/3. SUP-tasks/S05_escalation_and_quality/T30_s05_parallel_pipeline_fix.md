# T30_s05_parallel_pipeline_fix — Фикс параллельных pipeline (тройной ответ)

## Customer-facing инкремент

Клиент отправляет несколько сообщений подряд — получает один ответ на каждое, без дублей и гонок. Бот не присылает три ответа одновременно.

## Цель

Добавить per-chat asyncio.Lock в `_handle_authorized()` чтобы pipeline обрабатывал сообщения одного пользователя последовательно.

## Контекст

Вероятная причина тройного ответа в dialog2: клиент написал три сообщения подряд быстро («почему не знаете», «никто не отвечает», «за что я вам плачу»). Каждое запустило `SupportService.answer()` параллельно — все три завершились и каждый прислал ответ. Это race condition, а не баг Coordinator.

## Scope

- `handlers/text.py` — функция `_handle_authorized()`:
  - Добавить глобальный dict локов: `_chat_locks: dict[int, asyncio.Lock] = {}`
  - При входе в `_handle_authorized` получать или создавать лок для `chat_id`:
    ```python
    lock = _chat_locks.setdefault(chat_id, asyncio.Lock())
    async with lock:
        # весь существующий код _handle_authorized
    ```
  - Лок гарантирует: пока обрабатывается сообщение N для chat_id X — сообщение N+1 ждёт

- Cleanup локов: при очень большом числе пользователей dict может расти. Добавить удаление лока после завершения если `not lock.locked()` — или оставить без cleanup (приемлемо для текущего масштаба).

## Out of scope

- Дедупликация по `update_id` — уже реализована в middleware на уровне Supabase
- Изменение промптов или логики эскалации

## Технические детали

`asyncio.Lock` — корутинный примитив, не блокирует event loop. Пока первый pipeline работает (5-15 сек GPT-вызовы), второе сообщение ждёт в очереди. После завершения первого — обрабатывается второе.

Важно: лок оборачивает весь блок `_handle_authorized`, включая typing indicator и вызов `support_svc.answer()`.

Глобальный dict `_chat_locks` в модуле `handlers/text.py` — допустимо, т.к. бот single-process.

Диагностика до фикса: проверить логи на наличие параллельных вызовов — искать `SupportService.answer` с одинаковым `chat_id` и близкими timestamps.

## Как протестировать

### 1. Параллельные сообщения

1. Отправить 3 сообщения подряд очень быстро (в течение 1 секунды)
2. Проверить: ответы приходят последовательно (не одновременно пачкой)
3. Каждое сообщение получает один ответ

### 2. Нормальный диалог не деградирует

1. Написать сообщение, дождаться ответа, написать следующее
2. Проверить: поведение как раньше, нет дополнительных задержек

### 3. Typing indicator

1. Отправить сообщение
2. Проверить: typing indicator показывается пока обрабатывается первое, второе сообщение ждёт

## Критерии приёмки

1. 3 быстрых сообщения подряд → 3 последовательных ответа (не одновременно)
2. Нет дублирования ответов
3. Нет ошибок при конкурентной нагрузке
4. Обычный диалог работает без изменений
