# T04 — Обработка недоступности Bitrix

## Customer-facing инкремент

Если Bitrix временно недоступен, клиент видит понятное сообщение о технических проблемах — а не вводящий в заблуждение ответ "ИНН не найден".

## Цель

Разделить два разных сценария в `handlers/text.py`: "ИНН не найден в Bitrix" (штатная ситуация) и "Bitrix недоступен / вернул ошибку" (технический сбой). При сбое — отдельное сообщение, `error_count` не инкрементируется.

## Scope

- `handlers/text.py` (ветка `WAITING_INN`):
  - Обернуть вызов `bitrix.find_deal_by_inn()` в `try/except`
  - При исключении (таймаут, сетевая ошибка, HTTP 5xx) → отправить фиксированное сообщение о технических проблемах, **не** увеличивать `error_count`
  - Штатный `None` от Bitrix (ИНН не найден) — поведение без изменений
- `services/bitrix.py`:
  - Убедиться что `find_deal_by_inn()` выбрасывает исключение при сетевой ошибке (а не возвращает `None`)
  - Если сейчас возвращает `None` в обоих случаях — разделить через исключение

### Текст сообщения при сбое

```
Возникли временные технические проблемы. Попробуйте ещё раз через несколько минут.
```
Фиксированная строка, без обращения к OpenAI.

## Out of scope

- Retry-логика (повторные попытки запроса к Bitrix)
- Уведомления разработчика при сбое
- Обработка недоступности Bitrix в других сценариях (телефон и т.д.)

## Как протестировать

1. Временно сломать `BITRIX_WEBHOOK_BASE` в `.env` (неверный URL)
2. Ввести любой 12-значный ИНН
3. Проверить: бот отвечает сообщением о технических проблемах (не "ИНН не найден")
4. Проверить: `error_count` в Supabase не увеличился
5. Восстановить `BITRIX_WEBHOOK_BASE`, ввести несуществующий ИНН → поведение "не найден" (счётчик растёт)

## Критерии приёмки

1. При сетевой ошибке/таймауте Bitrix → клиент видит сообщение о технических проблемах
2. `error_count` при техническом сбое не увеличивается
3. Штатный сценарий "ИНН не найден" не затронут
