# T11 — Обогащение контекста клиента через Bitrix

## Customer-facing инкремент

Клиент получает ответ, основанный на реальном статусе его дела: ИИ знает текущую стадию процедуры и какие документы уже сданы — и не выдаёт общий чеклист тому, кто давно всё предоставил.

## Цель

Добавить в `BitrixService` метод `get_deal_profile(deal_id)`, который одним batch-запросом получает стадию сделки, имя ответственного менеджера и статус 16 документов из чеклиста. Объединить этот профиль с судебными документами из Supabase в единый `client_context`, передаваемый всем агентам.

## Изменения в коде

- `services/bitrix.py`:
  - Добавить константы `DOCUMENT_FIELDS` (список из 16 кортежей `(field_id, name)`) и `STAGE_LABELS` (маппинг `STAGE_ID → русское название`)
  - Добавить метод `get_deal_profile(deal_id: str) -> str`:
    - Batch-запрос: `crm.deal.get` (поля стадии + 16 документальных полей + `ASSIGNED_BY_ID`) + `user.get` (имя менеджера)
    - Форматировать строку с заголовком `=== ПРОФИЛЬ КЛИЕНТА (Bitrix) ===`, стадией, менеджером и чеклистом (✅ да / ⏳ ожидает, «нет» — не выводить)
    - При любой ошибке — вернуть `""` (не блокировать основной флоу)

- `services/support.py`:
  - Добавить параметр `deal_profile: str = ""` в `answer()`
  - Если `deal_profile` не пустой — prepend к `judicial_docs`: `client_context = f"{deal_profile}\n\n{judicial_docs}"`

- `handlers/text.py`:
  - В `_handle_authorized()`: добавить получение `deal_id = session.get("deal_id") or ""`
  - Перед вызовом `support_svc.answer()` вызвать `bitrix.get_deal_profile(deal_id)` (в try/except)
  - Передать `deal_profile` в `support_svc.answer()`

## Как протестировать

1. Авторизоваться с ИНН клиента, у которого в Bitrix есть сделка с заполненными полями документов
2. Спросить: «Какие документы мне нужно ещё сдать?»
3. Проверить: ответ опирается на фактический статус документов из Bitrix, а не выдаёт общий список
4. Спросить: «На какой стадии моё дело?»
5. Проверить: ответ называет реальную стадию (например, «Реализация имущества»), а не общую фразу

## Критерии приёмки

1. `DOCUMENT_FIELDS` и `STAGE_LABELS` присутствуют в `services/bitrix.py`
2. Метод `get_deal_profile()` возвращает форматированную строку с профилем или `""` при ошибке
3. `client_context` в агентах содержит Bitrix-профиль перед судебными документами
4. Если Bitrix недоступен — бот продолжает работу без Bitrix-профиля (graceful degradation)
5. Ответ ИИ отражает реальную стадию и статус документов клиента
