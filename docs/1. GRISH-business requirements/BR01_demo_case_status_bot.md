# BR01: Демо-бот «Что с моим делом?»

**Статус:** draft  
**Дата:** 2026-04-02  
**Автор:** Zhigalov  

---

## 1. Проблема и JTBD

Инвестор Гришин хочет оценить ценность AI-продукта для клиентов АрбитрА до принятия решения об инвестиции.

Клиент в процессе банкротства регулярно задаёт менеджеру сопровождения два вопроса: «Что происходит с моим делом?» и «Что мне нужно делать дальше?». Сейчас это требует участия живого менеджера. Friction: клиент не может получить ответ в любое время, менеджер тратит время на рутинные статусные обращения.

---

## 2. Как решают сейчас

Клиент пишет или звонит своему менеджеру сопровождения. Менеджер вручную смотрит в CRM и даёт ответ. Нет самообслуживания, нет мгновенного ответа, менеджер перегружен типовыми вопросами.

---

## 3. Что предлагаем

Отдельный Telegram-бот, который авторизует клиента по ИНН и телефону, находит его дело в базе и отвечает ровно на два вопроса:

- **«Что с моим делом?»** — структурированная сводка по делу: статус в суде, документальная готовность, финансовая часть (долги, платежи).
- **«Какие дальнейшие действия?»** — AI определяет конкретные следующие шаги клиента на основе текущего состояния дела и знания общей процедуры банкротства.

Бот не делает ничего лишнего — только эти два ответа, максимально точно и человечно.

---

## 4. Пользовательский флоу

1. Клиент открывает бота → вводит ИНН текстом → нажимает единственную кнопку «Поделиться номером» → авторизован.
2. Бот приветствует и предлагает задать вопрос.
3. Клиент пишет вопрос в свободной форме — бот определяет намерение и отвечает:
   - Если спрашивает о статусе дела → отдаёт структурированную сводку.
   - Если спрашивает что делать дальше → формирует список конкретных шагов.
4. Диалог продолжается — клиент может задавать оба вопроса в любом порядке и столько раз, сколько нужно.

Интерфейс полностью текстовый. Кнопок нет, кроме стандартной Telegram-кнопки «Поделиться номером» на этапе авторизации.

---

## 5. Метрики успеха

**Для демо:**
- Гришин видит, что бот знает конкретное дело клиента (точность данных)
- Ответы воспринимаются как структурированные, лаконичные, человечные
- Нет галлюцинаций — бот не выдумывает факты, которых нет в деле

**Долгосрочно:**
- Снижение доли статусных обращений к менеджеру
- Клиент понимает следующий шаг без звонка в офис

## 6. Описание (формализованное)

```
spec_type: demo_product / standalone_telegram_bot

product_context:
  Отдельный Telegram-бот для инвестора-демо. Работает на реальных данных из
  Supabase (electronic_case) и Bitrix24. Целевая аудитория — клиенты АрбитрА
  в процедуре банкротства физических лиц.

flow_and_solution:
  Авторизация: ИНН (12 цифр) текстом + кнопка «Поделиться номером» (единственная кнопка в боте).
  После авторизации — чистый диалог. Клиент пишет в свободной форме.
  Бот распознаёт намерение и отвечает:
    - Вопрос о статусе дела → сводка по делу
    - Вопрос о следующих шагах → пошаговые рекомендации

core_behavior:
  Вопрос 1 — «Что с моим делом?»:
    - Статус дела в суде (этап процедуры)
    - Документальная готовность (какие документы сданы / не сданы)
    - Финансовая часть (задолженности, платежи по делу)
    - Ответ структурирован, лаконичен, без канцелярита

  Вопрос 2 — «Дальнейшие действия?»:
    - AI анализирует текущее состояние дела
    - Формирует конкретный список шагов (что нужно сделать клиенту прямо сейчас)
    - Опирается на данные дела + знание процедуры банкротства физлиц
    - Не даёт общих советов — только то, что актуально для этого дела

data_model:
  Источник: Supabase electronic_case (cases, communications, checklist)
  Дополнительно: Bitrix24 (статус сделки, поля дела)
  Авторизация: Supabase bot_sessions (та же логика, что в основном боте)

conversation_principles:
  - Структура: короткие абзацы или списки, не сплошной текст
  - Тон: человечный, без бюрократии, уверенный
  - Точность: только факты из дела, без домыслов
  - Лаконичность: минимум слов, максимум смысла
  - ParseMode: HTML (<b>, <i>, <code>)

completion_conditions:
  - Тестировщик авторизуется за реального клиента
  - Бот корректно отображает данные конкретного дела
  - Оба ответа воспринимаются как точные и полезные
  - Гришин видит ценность продукта
```

## 7. Этапы раскатки

1. **Demo-среда** — отдельный бот-токен, работает на production-данных electronic_case.
2. **Тестирование** — прогон на 2–3 реальных делах перед показом.
3. **Показ Гришину** — тестировщик демонстрирует оба сценария вживую.

Масштабирование до реальных клиентов — отдельное решение после демо.

---

## 8. Подготовка поддержки и маркетинга

Для демо:
- Подобрать 2–3 дела с хорошо заполненными данными в electronic_case для убедительного показа.
- Убедиться, что данные по этим делам актуальны в Supabase перед демо.
