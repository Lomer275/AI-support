# Гайд: декомпозиция спецификаций на задачи

**Версия:** 1.0  
**Дата создания:** 22 ноября 2025  
**Статус:** Действует

---

## 🎯 Цель

Установить единый подход к декомпозиции спецификаций на задачи, обеспечивающий:
- Проверяемые инкременты на каждом шаге
- Управляемый размер задач (до 50% контекста GPT)
- Чёткую связь между спекой, задачами и customer-facing результатами
- Прозрачность прогресса и зависимостей

---

## 1. Принципы декомпозиции

### 1.1. Декомпозиция по инкрементам, а не по слоям

❌ **Плохо** (по слоям):
```
T01 — FSM состояния
T02 — Шаблоны сообщений
T03 — Модели БД
T04 — API endpoints
```

✅ **Хорошо** (по инкрементам):
```
T01 — Пользователь может создать проект через capture-flow
T02 — Пользователь может добавить медиа в первую запись
T03 — Админ видит ленту первых записей
```

**Почему:**
- Каждая задача даёт проверяемый результат
- Можно показать прогресс заказчику после каждой задачи
- Проще тестировать и принимать работу

### 1.2. Размер задачи: до 50% контекста GPT

**Ориентир:** задача должна быть решаема AI за один проход без переполнения контекста.

**Практические критерии:**
- 1-3 файла handlers/routers
- 1-2 модели данных или API endpoints
- До 10 новых шаблонов сообщений
- До 5 состояний FSM

**Если задача больше:**
- Разбить на подзадачи по инкрементам
- Выделить техническую подготовку (рефакторинг, миграции) в отдельную задачу

### 1.3. Каждая задача = проверяемый инкремент

**Обязательно:**
- Задача должна давать результат, который можно **показать** и **проверить**
- Результат должен быть **ценным** (пусть и маленький шаг к итоговой ценности спеки)

**Допустимые исключения:**
- Технический рефакторинг (но с явным обоснованием "зачем")
- Миграции данных (но с проверяемым результатом: "данные мигрированы корректно")

## 2. Подходы к декомпозиции

### 2.1. По user journey (для UX-тяжёлых фич)

**Пример:** Онбординг (S07)

Декомпозиция по шагам пользователя в онбординге: FSM capture-flow → шаблоны сообщений → сохранение сущностей → медиа в записи → admin feed.

**Задачи:**
- T35 — FSM capture-flow и маршрутизация (базовый флоу)
- T36 — Шаблоны сообщений (Jinja2)
- T37 — Сохранение первых сущностей (БД)
- T38 — Медиа в первой записи
- T39 — Admin feed (отдельный slice)

**Порядок выполнения:**
1. T35 (база, блокирует T36, T37)
2. T36 и T37 параллельно
3. T38 (требует T37)
4. T39 (требует T37, можно параллельно с T38)

**Полный пример:** см. раздел 2.1 "По user journey" и раздел 7.2 "Средняя фича (3-5 задач)"

### 2.2. По техническим слоям (для API-тяжёлых фич)

**Пример:** Backend API для TMA

```
Спека: API endpoints для TMA

Задачи:
T40 — Авторизация через Telegram initData + JWT
     Инкремент: Пользователь может авторизоваться в TMA через Telegram
     
T41 — API для работы с проектами (CRUD)
     Инкремент: Пользователь может создавать/редактировать проекты через TMA
     
T42 — API для работы с задачами (CRUD + фильтры)
     Инкремент: Пользователь может управлять задачами через TMA
     
T43 — API для работы с записями (создание + медиа)
     Инкремент: Пользователь может создавать записи с медиа через TMA
```

**Порядок выполнения:**
1. T40 (авторизация — блокирует всё остальное)
2. T41, T42, T43 последовательно (каждый зависит от предыдущего)

### 2.3. По независимым фичам (для параллельной работы)

**Пример:** Улучшения UX

Независимые задачи, которые не блокируют друг друга и могут выполняться параллельно: T44 (кнопка создания проекта), T45 (inline-выбор проекта/задачи), T46 (подтверждение сохранения с перевязкой).

**Порядок выполнения:**
- T44, T45, T46 можно делать параллельно (независимые изменения)

**Как нумеровать:** см. раздел 3.4 "Параллельные задачи"

### 2.4. Выделение технической подготовки

**Когда нужно:**
- Рефакторинг перед новой фичей
- Миграция данных
- Обновление зависимостей

**Пример:**

```
T47 — Рефакторинг FSM: вынести общие состояния в базовый класс
     Инкремент: Разработчик может переиспользовать общие состояния FSM
     Обоснование: Упрощает добавление новых FSM-сценариев в T48-T50
     
T48 — Новый FSM-сценарий: редактирование проекта
     Инкремент: Пользователь может редактировать название и описание проекта
```

**Правило:** Технические задачи должны иметь явное обоснование "зачем" и связь с последующими задачами.

## 3. Нумерация задач (TNN)

### 3.1. Принципы нумерации

- Нумерация **сквозная** по всему проекту (не привязана к номерам спек)
- Номера выдаются **осмысленно** — в порядке выполнения
- Номер задачи = её место в очереди разработки

### 3.2. Как определить следующий номер

**Через командную строку (PowerShell):**
```powershell
Get-ChildItem -Path "docs\3. Tasks" -Recurse -Filter "T*.md" | 
  ForEach-Object { $_.Name -replace "T(\d+)_.*", '$1' } | 
  Sort-Object -Descending | 
  Select-Object -First 1
```

**Вручную:**
1. Просмотреть все файлы в `docs/3. Tasks/` и `docs/3. Tasks/Done/`
2. Найти максимальный номер `TNN`
3. Использовать следующий номер (максимальный + 1)

### 3.3. Нумерация при декомпозиции спеки

**Правило:** Задачи одной спеки получают **последовательные номера** в порядке выполнения.

**Пример:**

Спека S07 декомпозируется на задачи T35-T39 (последовательная нумерация по порядку выполнения). Следующая спека S08 начинается с T40.

**Почему так:**
- Порядок номеров = порядок выполнения
- Легко понять зависимости по номерам
- Можно идти по задачам последовательно

**Полный пример:** см. раздел 2.1 "По user journey" и раздел 7.2 "Средняя фича"

### 3.4. Параллельные задачи

Если задачи можно делать параллельно, но они относятся к одной спеке:

**Вариант 1:** Нумеровать последовательно (T44, T45, T46), но в HANDOFF.md указать, что можно делать параллельно

**Вариант 2:** Использовать буквенные суффиксы (T44a, T44b, T44c) — только если задачи создаются одновременно

**Рекомендация:** Вариант 1 (последовательная нумерация + пометка в HANDOFF).

**Пример:** см. раздел 2.3 "По независимым фичам" (задачи T44-T46)

## 4. Структура задачи

### 4.1. Шапка (обязательно)

```markdown
Дата: YYYY-MM-DD  
Статус: draft | in progress | done  
Спецификация: docs/2. Specifications/SNN_<имя>.md  
Бизнес-требования: docs/1. businees requirements/<имя>.md (если есть прямая связь)

# TNN_<snake_case_название> — <Краткое описание>
```

**Правила:**
- Ссылка на спеку — **всегда первая строка** (после даты/статуса)
- Название задачи в заголовке — краткое, но понятное
- Статус обновляется по мере выполнения

### 4.2. Customer-facing инкремент (обязательно)

```markdown
## Customer-facing инкремент

Пользователь может <действие> и получает <результат>.

**Примеры:**
- Пользователь может создать проект через capture-flow и получить первую запись в системе.
- Пользователь может добавить фото/аудио в первую запись и увидеть подтверждение сохранения.
- Админ может открыть ленту записей и увидеть последние 50 записей с фильтрами.
```

**Формат:**
- Начинать с "Пользователь может..." или "Админ может..."
- Описывать **конечный результат**, а не технические детали
- Одно-два предложения, максимум абзац

**Почему это важно:**
- Сразу понятно, **зачем** эта задача
- Можно проверить результат без чтения всей задачи
- Связь с бизнес-ценностью очевидна

### 4.3. Контекст (для draft-задач)

```markdown
## Контекст

- Текущее состояние системы
- Проблема или потребность, которую решаем
- Наблюдения из фидбека пользователей (если есть)
```

**Когда нужен:**
- Задача решает конкретную проблему UX
- Есть фидбек от пользователей
- Нужно объяснить "почему сейчас"

**Когда не нужен:**
- Задача — очевидная часть спеки
- Контекст уже описан в спеке

### 4.4. Проблема (для draft-задач, опционально)

```markdown
## Проблема

- Что именно не работает или работает плохо
- Какие последствия для пользователя
- Почему текущее решение неудовлетворительно
```

**Используется для:**
- Багфиксов
- Улучшений UX
- Задач, возникших из фидбека

### 4.5. Цель (для draft-задач)

```markdown
## Цель

Сделать так, чтобы <описание желаемого поведения>.

**Пример:**
Сделать так, чтобы по нажатию на кнопку «Создать проект» пользователь 
однозначно понимал, что бот ждёт ввода названия, и видел это состояние 
без ручного скролла и дополнительных догадок.
```

**Отличие от инкремента:**
- Инкремент — что **получает** пользователь
- Цель — что **меняется** в поведении системы

### 4.6. Scope / Out of scope (обязательно для draft)

```markdown
## Scope

- Конкретный перечень изменений, экранов, компонентов
- Какие файлы/модули затрагиваются
- Какие сценарии покрываются

## Out of scope

- Что точно не делаем в этой задаче
- Что откладываем на будущее
- Что может отвлечь от основной цели
```

**Почему важно:**
- Ограничивает размер задачи
- Предотвращает scope creep
- Помогает AI не уходить в сторону

### 4.7. Требования к поведению (для draft-задач)

```markdown
## Требования к поведению

### Базовый сценарий <название>

- Предусловия: ...
- Шаги:
  1. ...
  2. ...
- Ожидаемое поведение: ...

### Возможные варианты реализации

- Вариант 1: ...
- Вариант 2: ...
```

**Когда нужен:**
- Задача затрагивает UX/UI
- Есть несколько способов реализации
- Нужно зафиксировать ожидания до разработки

### 4.8. Изменения в коде (для done-задач)

```markdown
## Изменения в коде

- `bot/handlers/onboarding.py`:
  - Добавлена функция `show_home_screen()`
  - Обновлён обработчик `/start` с проверкой `first_entry_done`
- `shared/models.py`:
  - Добавлено поле `recent_task_ids` в `UserProfile`
- `shared/db.py`:
  - Реализован `update_user_last_task_id()` с LRU-списком
```

**Формат:**
- Группировка по файлам
- Краткое описание изменений (не diff, а суть)
- Упоминание новых функций/классов/полей

### 4.9. Как протестировать (обязательно)

```markdown
## Как протестировать

### 1. Базовый сценарий

1. Шаг 1
2. Шаг 2
3. Проверить:
   - что A работает
   - что B отображается
   - что C сохраняется

### 2. Негативный сценарий

1. ...
2. Проверить, что ошибка обрабатывается корректно

### 3. Регресс

1. ...
2. Проверить, что старая функциональность не сломалась
```

**Обязательно для:**
- Draft-задач (чтобы понять, как проверять результат)
- Done-задач (чтобы воспроизвести проверку)

**Формат:**
- Пошаговые инструкции
- Конкретные команды (если нужны)
- Ожидаемые результаты

### 4.10. Критерии приёмки / DoD (обязательно)

```markdown
## Критерии приёмки

1. Пользователь может создать проект через capture-flow.
2. После сохранения записи показывается главное меню.
3. Повторный `/start` открывает домашний экран.
4. Логи фиксируют создание всех сущностей.
5. Автоматические тесты, относящиеся к этой задаче, проходят без ошибок.
```

**Формат:**
- Нумерованный список, чтобы на отдельные пункты было удобно ссылаться (например, «см. критерий 3»).
- Каждый пункт — отдельный, чётко проверяемый факт (функциональный или технический).
- Список должен быть коротким и исчерпывающим для данной задачи.

**Отличие от DoD спеки:**
- DoD спеки = итоговая ценность для пользователя
- DoD задачи = проверяемый шаг к этой ценности

### 4.11. Связанные документы (опционально)

```markdown
## Связанные документы

- Спека: `docs/2. Specifications/S07_onboarding_capture_flow.md`
- Бизнес-требования: `docs/1. businees requirements/03_releases.md` (R1, Этап 1)
- Релиз: `docs/releases/02-core-mvp.md`
```

**Примечание:** Правила пересборки контейнеров и завершения задачи описаны в разделе 5 "Жизненный цикл задачи" (подразделы 5.4 и 5.3 соответственно). Эти разделы должны быть включены в каждую задачу.

## 5. Жизненный цикл задачи

### 5.1. Создание (draft)

```markdown
Дата: 2025-11-22
Статус: draft
Спецификация: docs/2. Specifications/S08_tma_launcher_bot.md

# T50_simple_bot_for_tma_launch — Простой бот для запуска TMA

## Customer-facing инкремент

Я могу отправить боту данные (object_id + areas/items) и получить ссылку для запуска TMA.

## Цель

Создать временный бот-костыль для быстрого запуска TMA с предзаполненными данными.

## Scope

- Простой Telegram-бот (aiogram)
- Диалог для ввода object_id, areas, items
- Генерация ссылки/кнопки для запуска TMA с JSON в start_param

## Out of scope

- Интеграция с основным backend (bot/)
- Хранение данных в БД
- Полноценный lifecycle объектов

## Как протестировать

1. Запустить бота
2. Ввести object_id, areas, items
3. Получить ссылку для запуска TMA
4. Открыть TMA и проверить, что данные передались

## Критерии приёмки

- [ ] Бот принимает данные через диалог
- [ ] Генерируется корректная ссылка для TMA
- [ ] Данные передаются в TMA через start_param
- [ ] Есть README с инструкцией по использованию

## Пересборка контейнеров после выполнения

*См. раздел 5.4 для полного описания.*

## Правила завершения задачи

*См. раздел 5.3 для полного описания.*
```

### 5.1.1. Внесение в HANDOFF.md (ОБЯЗАТЕЛЬНО)

**При создании новой задачи ОБЯЗАТЕЛЬНО добавить её в HANDOFF.md:**

1. **Спросить у пользователя приоритет** — предложить варианты:
   - "Приоритет 1 (первым в списке) — hotfix/critical"
   - "Приоритет 1 (в конец списка) — обычная задача текущего приоритета"
   - "Приоритет 2/3/Backlog — отложить"
2. **Дождаться ответа** — не додумывать самостоятельно
3. **Добавить в секцию** согласно ответу пользователя
4. **Указать статус** `draft`

**Hotfix/critical задача:**
- Добавить **в начало** списка соответствующего приоритета
- Выделить **жирным**

**Обычная задача:**
- Добавить в конец списка соответствующего приоритета

**Пример добавления в HANDOFF.md:**

```markdown
### Приоритет 1: S25 — Retry Flow

| ID | Задача | Статус |
|----|--------|--------|
| [T202](docs/3.%20Tasks/...) | **Upload race condition fix** | draft |  ← hotfix, первый
| [T200](docs/3.%20Tasks/...) | Retry Flow: processing vs actionable | draft |
```

**ВАЖНО для AI:** Создание файла задачи БЕЗ добавления в HANDOFF.md — незавершённая работа. Всегда делать оба действия.

### 5.2. В работе (in progress)

```markdown
Статус: in progress
```

- Обновить статус в файле задачи
- Обновить HANDOFF.md (пометить задачу как "в работе")

### 5.3. Завершение (done)

**ВАЖНО:** Перенос задачи в DONE и все сопутствующие изменения выполняются **ТОЛЬКО после явного акцепта** от пользователя (формулировки: "задача акцептована", "принято", "task accepted" и т.п.).

**После акцепта задачи AI выполняет АВТОМАТИЧЕСКИ (без вопросов):**

1. **Обновить файл задачи:**
   - Изменить статус на `✅ accepted` или `done`
   - Добавить дату завершения в формате `YYYY-MM-DD`
   - Переписать разделы на прошедшее время (если нужно)

2. **Переместить файл задачи:**
   - Из: `docs/3. Tasks/SNN_*/TNN_*.md`
   - В: `docs/3. Tasks/Done/SNN_*_done/TNN_*_done.md`
   - Переименовать с суффиксом `_done.md`

3. **Обновить ссылки в спецификации:**
   - Ссылки на задачи → новые пути в Done
   - Статусы → `✅ accepted`

4. **Обновить HANDOFF.md:**
   - В разделе "Текущее состояние": добавить запись о завершённой задаче
   - В разделе "Следующие шаги": пометить задачу как выполненную
   - Обновить связанные секции (техдолг, приоритеты и т.п.)

5. **Обновить CHANGELOG.md:**
   - Добавить запись под секцией `[Не выпущено] → Добавлено`:
     ```markdown
     - YYYY-MM-DD — TNN: <краткое описание инкремента>
     ```

6. **Пересборка контейнеров** (если были изменения кода)

**Правило для AI-агента:**
- НЕ перемещать задачу в DONE без явного акцепта пользователя
- НЕ обновлять HANDOFF и CHANGELOG до акцепта
- **После акцепта — выполнить ВСЁ автоматически, БЕЗ вопросов и уточнений**

### 5.3.1. Шаблон раздела "Правила завершения задачи"

**Этот раздел ОБЯЗАТЕЛЬНО копировать в каждую задачу:**

```markdown
## Правила завершения задачи

**После акцепта задачи AI выполняет автоматически:**

1. **Обновить файл задачи:**
   - Статус → `✅ accepted`
   - Добавить дату завершения

2. **Переместить файл:**
   - Из: `docs/3. Tasks/SNN_*/TNN_*.md`
   - В: `docs/3. Tasks/Done/SNN_*_done/TNN_*_done.md`

3. **Обновить спецификацию:**
   - Ссылки на задачи → новые пути
   - Статусы → обновить

4. **Обновить CHANGELOG.md:**
   - Добавить запись в секцию `[Не выпущено] → Добавлено`

5. **Обновить HANDOFF.md:**
   - Отметить задачу как выполненную
   - Обновить связанные секции

6. **Пересобрать контейнеры** (если были изменения кода)
```

**Важно:** Без этого раздела задача считается неполной.

### 5.4. Пересборка контейнеров после выполнения

**Когда требуется:**
- Всех задач, которые вносят изменения в код, работающий в Docker контейнерах
- Задач, затрагивающих backend (`bot/`)
- Задач, затрагивающих фронтенд (`TMA_functional/`)
- Задач, изменяющих конфигурацию Docker (`Dockerfile`, `docker-compose.yml`)

**Когда не требуется:**
- Задачи, которые только добавляют/изменяют документацию
- Задачи, которые только обновляют спецификации без изменения кода

**Процесс:**

**Важно:** AI должен автоматически пересобрать затронутые контейнеры после завершения всех изменений в коде.

**AI должен выполнить:**
```bash
# Пересобрать все контейнеры (рекомендуется)
docker-compose build

# Или пересобрать только конкретные сервисы, если изменения затрагивают только их:
docker-compose build bot
docker-compose build tma_functional
```

После пересборки перезапустить сервисы:
```bash
docker-compose up -d
```

**Примечание:** Это должно быть выполнено автоматически AI в конце работы над задачей, без запроса пользователя.

**Важно:** Этот раздел должен быть включён в каждую задачу с изменениями кода (см. пример в разделе 5.1).

### 5.5. Отмена (cancelled)

Если задача отменена (решение пересмотрено, заморожено):

1. Переименовать файл с суффиксом `_cancelled.md`
2. Добавить в начало файла:
   ```markdown
   **Статус:** ОТМЕНЕНО  
   **Дата отмены:** 2025-11-22  
   **Причина:** Функционал признан избыточным для MVP, перенесен в backlog
   ```
3. Переместить в `docs/3. Tasks/Done/SNN_*_done/`
4. Обновить HANDOFF.md и CHANGELOG.md

## 6. Управление зависимостями и порядком

### 6.1. Где хранить зависимости

**Правило:** Зависимости и порядок выполнения ведутся в `HANDOFF.md`, а не в файлах задач.

**Почему:**
- Единая точка истины для планирования
- Не нужно обновлять ссылки в задачах при переименовании
- Легко перестроить порядок при изменении приоритетов

### 6.2. Формат в HANDOFF.md

```markdown
## ✅ Следующие шаги

### Приоритет 1: Онбординг (S07)

**Порядок выполнения:**
1. T35 — FSM capture-flow и маршрутизация /start
   - **Блокирует:** T36, T37
   - **Инкремент:** Пользователь может пройти capture-flow
   
2. T36 — Шаблоны сообщений (Jinja2)
   - **Требует:** T35
   - **Можно параллельно с:** T37
   - **Инкремент:** Пользователь видит понятные тексты на каждом шаге
   
3. T37 — Сохранение первых сущностей
   - **Требует:** T35
   - **Можно параллельно с:** T36
   - **Блокирует:** T38
   - **Инкремент:** Данные сохраняются в БД
   
4. T38 — Медиа в первой записи
   - **Требует:** T37
   - **Инкремент:** Пользователь может добавить фото/аудио
   
5. T39 — Admin feed
   - **Требует:** T37 (нужны данные в БД)
   - **Можно параллельно с:** T38
   - **Инкремент:** Админ видит ленту записей
```

### 6.3. Типы зависимостей

- **Блокирует:** Эта задача должна быть завершена до начала следующих
- **Требует:** Эта задача не может начаться без завершения предыдущих
- **Можно параллельно с:** Эти задачи независимы и могут выполняться одновременно

## 9. Частые ошибки

### ❌ Задача слишком большая

**Признаки:**
- Затрагивает 5+ файлов handlers/routers
- Требует изменений в 3+ моделях данных
- Описание занимает больше 2 страниц

**Решение:** Разбить на подзадачи по инкрементам

### ❌ Задача не даёт проверяемого результата

**Признаки:**
- Инкремент описан как "Реализован рефакторинг..."
- Нет понятного способа проверить результат
- Нельзя показать заказчику

**Решение:** Переформулировать инкремент или выделить техническую подготовку в отдельную задачу с явным обоснованием

### ❌ Задача дублирует спеку

**Признаки:**
- Задача копирует разделы из спеки
- Нет уточнений и конкретики
- Непонятно, чем задача отличается от спеки

**Решение:** Задача должна быть **конкретнее** спеки: указывать файлы, функции, конкретные шаги реализации

### ❌ Нет связи с customer-facing результатом

**Признаки:**
- Описание только технических деталей
- Непонятно, зачем это пользователю
- Нет раздела "Customer-facing инкремент"

**Решение:** Добавить раздел с описанием, что получает пользователь

### ❌ Неясные зависимости

**Признаки:**
- Задача требует результатов другой задачи, но это не указано
- Порядок выполнения неочевиден
- Задачи конфликтуют (меняют одни и те же файлы)

**Решение:** Явно указать зависимости в HANDOFF.md, выстроить порядок

## 10. Локализация в задачах

Если задача добавляет или изменяет UI-тексты (cleaner-facing), необходимо обновить локализацию.

### 10.1. Когда нужно

- Добавляется новая кнопка, заголовок, сообщение об ошибке
- Меняется существующий текст
- Добавляется новый экран

### 10.2. Что делать в задаче

1. **В коде:** использовать `t('screen.element')` вместо хардкода
   ```tsx
   // ❌ Плохо
   <Button>Начать уборку</Button>

   // ✅ Хорошо
   <Button>{t('index.btn_start')}</Button>
   ```

2. **В Google Sheet:** добавить строку с ключом и английским текстом
   | key | en | th |
   |-----|----|----|
   | index.btn_start | Start cleaning | |

3. **Синхронизировать:** `npm run i18n:sync`

4. **В задаче:** добавить раздел "Локализация"
   ```markdown
   ## Локализация

   Добавлены ключи:
   - `index.btn_start` — "Start cleaning"
   - `index.error_no_data` — "No data available"

   Google Sheet обновлён, тайские переводы pending.
   ```

### 10.3. Формат ключей

```
{screen}.{element_type}_{description}
```

Примеры:
- `index.btn_start` — кнопка на экране Index
- `video_recording.error_camera` — ошибка на экране VideoRecording
- `completion.heading_done` — заголовок на экране Completion

### 10.4. Чек-лист для задач с UI

- [ ] Все новые тексты используют `t('key')`
- [ ] Ключи добавлены в Google Sheet с английским текстом
- [ ] Запущен `npm run i18n:sync`
- [ ] В задаче есть раздел "Локализация" со списком новых ключей
