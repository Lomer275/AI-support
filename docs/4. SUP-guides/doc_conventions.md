**Версия:** 1.0  
**Дата создания:** 21 ноября 2025  
**Статус:** Действует

## 🎯 Цель

Установить единые правила именования, структуры и размещения документов проекта для обеспечения:
- Прозрачности и порядка в документации
- Упрощения навигации и поиска
- Согласованности между задачами и спецификациями
- Четкого отслеживания истории изменений

## 1. Именование спецификаций

### Формат
```
SNN_<snake_case_название>.md
```

### Правила
- `NN` — двузначный номер (01, 02, 03, ..., 99)
- Нумерация **хронологическая** (увеличивается с новыми спецификациями)
- Название в `snake_case` (только латиница, строчные буквы, подчеркивания)
- Файл размещается в `docs/2. Specifications/` (или в подпапках API/UI)

### Примеры
```
S01_dual_bot_architecture.md
S02_video_processing_pipeline.md
S03_openai_integration.md
S04_airtable_dropbox_storage.md
```

### Статусы спецификаций

Спецификация может находиться в одном из статусов:
- **draft** — черновик, в разработке
- **active** — действующая, по ней ведется работа
- **done** — завершенная, все задачи выполнены
- **deprecated** — устаревшая, заменена другой спецификацией

Статус указывается в шапке документа:
```markdown
**Статус:** active
```

При завершении спецификации:
1. Изменить статус на `done` и добавить дату завершения
2. Обновить ссылки в HANDOFF.md и CHANGELOG.md
3. Опционально: добавить суффикс `_done` к имени файла (например, `S01_dual_bot_architecture_done.md`)

## 2. Именование задач

### Формат
```
TNN_<snake_case_название>.md
```

### Правила
- `NN` — двузначный номер (01, 02, 03, ..., 99)
- Нумерация **сквозная** по всем задачам (в порядке создания)
- Название в `snake_case` (только латиница, строчные буквы, подчеркивания)
- **Обязательно** в начале каждой задачи добавить строку:
  ```markdown
  **Спецификация:** docs/2. Specifications/SNN_<имя>.md
  ```

### Примеры
```
T01_pyrogram_userbot_setup.md
T02_aiogram_bot_setup.md
T03_database_models_sqlalchemy.md
T04_video_download_handler.md
```

## 3. Определение следующего номера задачи (TNN)

**Важно:** `CHANGELOG.md` не является источником истины для нумерации задач — он содержит только часть упомянутых задач.

### Правильный способ определения следующего номера:

**Через командную строку (PowerShell):**
```powershell
# Найти все файлы задач
Get-ChildItem -Path "docs\3. Tasks" -Recurse -Filter "T*.md" | 
  ForEach-Object { $_.Name -replace "T(\d+)_.*", '$1' } | 
  Sort-Object -Descending | 
  Select-Object -First 1
```

**Вручную:**
1. Просмотреть все файлы в `docs/3. Tasks/` и подпапках `Done/`
2. Найти максимальный номер `TNN`
3. Использовать следующий номер (максимальный + 1) для новой задачи

### Пример
- Если максимальный номер: `T08`
- Следующая задача получает номер: `T09`

### Для AI-ассистентов
- При создании новой задачи **всегда** выполнять поиск максимального номера среди всех файлов в `docs/3. Tasks/` (включая `Done/`)
- **Не полагаться** на `CHANGELOG.md` для определения следующего номера
- Проверять на конфликты: убедиться, что выбранный номер не используется

## 4. Размещение задач

### Текущие задачи (в работе)
```
docs/3. Tasks/TNN_<snake_case>.md
```

Новые задачи создаются в корне папки `docs/3. Tasks/`.

### Завершенные задачи
```
docs/3. Tasks/Done/SNN_<имя>_done/TNN_<snake_case>_done.md
```

**Правила:**
- Каждая группа задач, относящихся к спецификации `SNN_<имя>.md`, складывается в папку:
  `docs/3. Tasks/Done/SNN_<имя>_done/`
- Завершенная задача переименовывается с суффиксом `_done.md`
- В начале файла **обязательно** добавить ссылку на спецификацию

**Пример структуры:**
```
docs/3. Tasks/Done/
├── S01_dual_bot_architecture_done/
│   ├── T01_pyrogram_userbot_setup_done.md
│   ├── T02_aiogram_bot_setup_done.md
│   └── T03_graceful_shutdown_done.md
├── S02_video_processing_done/
│   ├── T04_video_download_handler_done.md
│   ├── T05_ffmpeg_audio_extraction_done.md
│   └── T06_whisper_transcription_done.md
```

### Отменённые задачи
```
docs/3. Tasks/Done/SNN_<имя>_done/TNN_<snake_case>_cancelled.md
```

**Правила:**
- Если выполнение задачи прекращено (решение пересмотрено/заморожено), файл переносится с суффиксом `_cancelled.md`
- В `HANDOFF.md` помечается как «ОТМЕНЕНО» с ссылкой на соответствующий файл
- Внутри файла кратко указать **причину отмены** и **дату**

**Пример:**
```markdown
**Статус:** ОТМЕНЕНО  
**Дата отмены:** 21 ноября 2025  
**Причина:** Функционал признан избыточным для MVP, перенесен в backlog
```

## 5. Единый бэклог

### Расположение
```
docs/backlog/
```

### Правила
- Единственная папка бэклога: `docs/backlog/`
- Перемещение в бэклог выполняется только по прямому распоряжению владельца продукта
- Допустимые элементы:
  - Спеки: `SNN_...` или идеи без номера (описательные `snake_case`)
  - Инициативы/наброски: описательные `snake_case`
  - Идеи для будущих фич: произвольные названия

### Активация из бэклога

**Спецификации:**
1. Переместить в `docs/2. Specifications/SNN_<имя>.md` (назначить номер)
2. Обновить ссылки во всех документах
3. Удалить исходник из `docs/backlog/`

**Инициативы:**
1. Оформить как задачи и переместить в `docs/3. Tasks/`
2. Либо по итогу выполнения в `docs/3. Tasks/Done/...`
3. Обновить ссылки, удалить исходник из `docs/backlog/`

**Проверки:**
- `grep` по старому пути должен быть пустым
- Обновить `CHANGELOG.md` (секция "Не выпущено")
- При необходимости обновить `HANDOFF.md` и `README.md`

## 6. Процедура переноса задачи (обязательно)

### Шаги

1. **Перенести файл** задачи в целевую папку:
   ```
   docs/3. Tasks/Done/SNN_<имя>_done/TNN_<snake_case>_done.md
   ```

2. **Добавить первой строкой** ссылку на спецификацию:
   ```markdown
   **Спецификация:** docs/2. Specifications/SNN_<имя>.md
   ```

3. **Удалить исходные черновики** и старые файлы:
   - Удалить файл из `docs/3. Tasks/`
   - Удалить пустые директории

4. **Обновить ссылки** во всём репозитории на новый путь задачи:
   - Поиск по старому имени файла
   - Замена на новый путь

5. **Обновить ключевые документы** при необходимости:
   - `CHANGELOG.md` (раздел «Не выпущено» — Добавлено/Изменено/Исправлено)
   - `HANDOFF.md` (актуальные ссылки, обновить "Текущее состояние")
   - `README.md` (если есть прямые ссылки)

6. **Проверка перед закрытием:**
   - Поиск по репозиторию по старому имени файла/пути — результатов быть не должно
   - Поиск по заголовку задачи — ссылки должны вести на новый путь

### Мини-чеклист к коммиту/PR

- [ ] Старый файл удалён
- [ ] Ссылки обновлены (CHANGELOG/HANDOFF/README/внутри задач)
- [ ] Grep по старому пути — пусто
- [ ] Реестр TNN актуализирован (если ведется)
- [ ] Коммит: `docs: apply conventions (remove old, update refs)`

## 7. Ссылки в ключевых документах

### README.md
- Содержит ссылку на этот гайд
- Содержит ссылки на HANDOFF.md и CHANGELOG.md

### HANDOFF.md
- Содержит ссылку на этот гайд
- Ссылается на `docs/backlog/` как единый источник бэклога
- Содержит список текущих задач со ссылками

### CHANGELOG.md
- Отражает структурные изменения документации в разделе «Изменено»
- Содержит ссылки на завершенные задачи

## 8. Правила для AI-ассистентов

### При создании новой задачи:
1. Всегда выполнять поиск максимального номера `TNN` среди всех файлов в `docs/3. Tasks/` (включая `Done/`)
2. Не полагаться на `CHANGELOG.md` для определения следующего номера
3. Проверять на конфликты: убедиться, что выбранный номер не используется в других спецификациях

### При завершении задачи:
1. Перенести файл в `docs/3. Tasks/Done/SNN_*_done/` с суффиксом `_done.md`
2. Добавить ссылку на спецификацию первой строкой
3. Удалить исходный файл из `docs/3. Tasks/`
4. Обновить ссылки во всех документах
5. Обновить CHANGELOG.md (секция "Не выпущено")
6. Обновить HANDOFF.md (секция "Текущее состояние")

### При создании спецификации:
1. Назначить следующий номер `SNN` (проверить максимальный в `docs/2. Specifications/`)
2. Указать статус (`draft`, `active`, `done`, `deprecated`)
3. Добавить дату создания и автора
4. Связать со спецификацией все связанные задачи

---

## 9. Общие правила

- **Всегда используйте текущую максимальную `NN`** и увеличивайте на 1 для новых файлов
- **При переносе/рефакторинге:** не меняйте смысловые части названий, только префиксы
- **В задачах ссылка на спецификацию обязательна**
- **Нумерация сквозная:** номера задач не привязаны к номерам спецификаций
- **Не дублируйте историю:** HANDOFF.md — текущее состояние, CHANGELOG.md — полная история
