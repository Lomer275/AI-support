## 🎯 Цель

На презентации 2026-03-26 ИИ-агент «Алина» показал неприемлемое качество ответов: галлюцинировал факты о компании (несуществующий офис в Ростове), игнорировал реальный статус дела клиента, повторно запрашивал уже предоставленные документы, не справлялся с конфликтными ситуациями.

Цель спринта — устранить корневые причины деградации качества и выстроить систему, при которой:
- ИИ отвечает **конкретно по делу клиента**, а не общими фразами
- ИИ **никогда не придумывает** факты о компании, которых нет в базе знаний
- ИИ **корректно ведёт диалог** с учётом всей истории беседы
- ИИ **распознаёт конфликтные ситуации** и мягко переключает на специалиста
- Качество ответов можно **объективно измерять и тестировать**

## 🔍 Диагностика: корневые причины сбоев

По диалогу с реальным клиентом (2026-03-26) выявлено 4 класса ошибок:

### А) Галлюцинации о компании
**Симптом:** ИИ предложил "обратиться в офис в Ростове-на-Дону" — офиса там нет уже давно.
**Корневая причина:** ни один из 7 агентов не имеет в промпте актуальных фактов о компании. Модель "вспоминает" устаревшие данные из обучающей выборки.
**Фикс:** Phase 1 — `COMPANY_FACTS`.

### Б) Игнор статуса дела клиента
**Симптом:** Клиент уже на стадии реализации имущества, все документы сданы — ИИ выдал общий чеклист "что нужно подготовить к банкротству".
**Корневая причина:** `client_context` содержит только судебные документы из support Supabase. Статус документов, стадия сделки в Bitrix, наличие/отсутствие пендингов — туда не попадают.
**Фикс:** Phase 2 — обогащение контекста через Bitrix.

### В) Потеря контекста беседы
**Симптом:** Клиент сказал "я всё предоставил!" — следующие ответы по-прежнему говорили о документах.
**Корневая причина:** координатор получает `history` (последние 10 сообщений), но агенты R1/R2 историю **не видят** — каждый обрабатывает только `client_context` + текущий вопрос. Синтез истории происходит только на этапе координатора, которому уже нужно собрать финальный ответ.
**Фикс:** Phase 3 — инжект истории в агенты R1/R2.

### Г) Отсутствие обработки конфликта
**Симптом:** При агрессии клиента ("верните деньги, бездельники!") ИИ ответил "Подождите несколько минут, мне нужно уточнить" и завис.
**Корневая причина:** `switcher=true` триггерится корректно, но сообщение-заглушка слишком нейтральное и не деэскалирует конфликт.
**Фикс:** Phase 3 — улучшенный шаблон эскалации.

### Phase 3: Доработка промптов

Два изменения в `support.py`:

1. **Инжект истории в R1/R2 агентов** — передавать `history_text` (аналогично тому, как это сделано у координатора) в промпты R1-юриста и R1-менеджера. Это самые критичные точки: юрист должен знать, что клиент уже сообщил ранее.

2. **Улучшенный шаблон `switcher=true`** — вместо нейтрального "Подождите несколько минут":

```
При switcher=true и признаках конфликта (агрессия, требование возврата, угрозы):
"Понимаю вашу озабоченность. Я передаю ваш запрос специалисту — он свяжется с вами в ближайшее время."

При switcher=true без конфликта (просьба связаться, вопрос про оплату):
"Этот вопрос требует участия специалиста — я передала запрос, вам скоро ответят."
```

Координатор должен различать эти два случая через дополнительный параметр в JSON: `"escalation_type": "conflict" | "request"`.

### Phase 4: ИИ-симулятор клиента

`SimulatorService` — отдельный агент OpenAI, получает:
- `client_profile` — стадия дела, эмоциональный тип клиента (тревожный / агрессивный / доверяющий)
- `scenario` — конкретный вопрос из `questions.md`
- `chat_history` — предыдущие ходы симуляции

Возвращает: следующее сообщение клиента в диалоге (учитывает полученный ответ ИИ).

Позволяет прогонять многоходовые сценарии — не просто Q→A, а полноценный диалог из 5-10 ходов.

### Phase 5: Скрипт оценки качества

`scripts/quality_run.py` — CLI:

```bash
python scripts/quality_run.py --questions docs/5.\ SUP-unsorted/questions.md --limit 20 --output results.json
```

Логика:
1. Берёт N вопросов из `questions.md`
2. Для каждого: запускает `SupportService.answer()` с тестовым `chat_id` и тестовым `inn`
3. Передаёт Q+A в `EvaluatorService`
4. Сохраняет оценку в `results.json`

`EvaluatorService` оценивает ответ по 4 критериям (0–3 балла каждый):
- **Специфичность** — ответ использует данные дела или общий?
- **Фактологическая точность** — нет галлюцинаций о компании?
- **Тональность** — успокаивает или нагнетает тревогу?
- **Полнота** — отвечает на весь вопрос или частично?

### Phase 5: Эскалация на живого оператора (Bitrix Open Lines)

#### Общая схема потока данных

```
Клиент (Telegram)
      │ switcher=true
      ▼
[handlers/text.py] ── session.escalated = True
      │
      ▼
[ImConnectorService.send_escalation()]
      │  POST /imconnector.send.messages.json
      │  OAuth access_token (Redis → .env)
      │  CONNECTOR = "arbitrasupportbot"
      │  chat.id = telegram_chat_id
      │  message = история диалога + вопрос
      │
      ▼
[Bitrix24 Open Lines]
      │  session.CHAT_ID сохраняется в bot_sessions
      │  imopenlines.operator.transfer → ответственный по сделке
      │
      ▼ (оператор отвечает)
[POST /webhook/bitrix/] ← Bitrix шлёт webhook
      │  data[CONNECTOR] = arbitrasupportbot
      │  data[MESSAGES][0][chat][id] = telegram_chat_id
      │
      ▼
[aiohttp WebApp (webhook_server.py)]
      │  найти сессию по chat_id, переслать текст в Telegram
      │
      ▼
Клиент видит ответ оператора
```

#### Новые файлы

```
services/
└── imconnector.py      # ImConnectorService — OAuth + отправка в Open Lines

webhook_server.py       # aiohttp web.Application — приём webhook от Bitrix
```

Webhook-сервер запускается **в том же процессе** что и aiogram polling:
- `bot.py` создаёт два корутина: `dp.start_polling(bot)` + `aiohttp.web.run_app(webhook_app)`
- Порт: `WEBHOOK_PORT` (env, default: 8080)
- Публичный URL: `WEBHOOK_HOST` (env, например `https://89.223.125.143:8080`) — должен быть доступен из интернета для Bitrix

#### ImConnectorService (`services/imconnector.py`)

```python
class ImConnectorService:
    async def send_escalation(
        self,
        telegram_chat_id: int,
        contact_name: str,
        phone: str,
        question: str,
        history: list[dict],
    ) -> str | None:  # возвращает bitrix_session_chat_id или None
```

**OAuth-авторизация:**
- Токены хранятся в `.env`: `BITRIX_OAUTH_ACCESS_TOKEN`, `BITRIX_OAUTH_REFRESH_TOKEN`, `BITRIX_OAUTH_CLIENT_ID`, `BITRIX_OAUTH_CLIENT_SECRET`
- Перед каждым вызовом: проверить TTL access_token (или поймать 401) → вызвать `refresh_tokens()`
- `POST https://oauth.bitrix24.tech/oauth/token/` с `grant_type=refresh_token`
- Обновлённые токены записываются обратно в `.env` (или во временный файл `oauth_tokens.json` рядом с `bot.py`)

**Тело запроса к ImConnector:**
```json
{
  "CONNECTOR": "arbitrasupportbot",
  "LINE": "<BITRIX_OPENLINE_ID>",
  "MESSAGES": [{
    "user":    { "id": "<telegram_chat_id>", "name": "<contact_name>", "phone": "<phone>" },
    "message": { "id": "0", "date": <unix_ts>, "text": "<история + вопрос>" },
    "chat":    { "id": "<telegram_chat_id>", "url": "" }
  }],
  "auth": "<oauth_access_token>"
}
```

**Формат текста эскалации:**
```
=== ИСТОРИЯ ДИАЛОГА С ИИ-БОТОМ ===
Клиент: <сообщение>
Алина: <ответ>
...
=== КОНЕЦ ИСТОРИИ ===
💬 Вопрос требует участия специалиста: <последний вопрос клиента>
```

**CRM-активность после эскалации** (`crm.activity.add`):
- `OWNER_TYPE_ID: 2` (сделка), `OWNER_ID: deal_id`
- `SUBJECT: "Запрос клиента — требует специалиста — <contact_name>"`
- `DESCRIPTION`: последние 5 сообщений истории

**Перевод на ответственного** (3-попытки, аналог оригинала):
1. `imopenlines.operator.transfer` — стандартный перевод
2. `imopenlines.bot.session.transfer` — с `LEAVE: Y`
3. fallback: `im.notify.system.add` — IM-уведомление ответственному

#### Webhook-сервер (`webhook_server.py`)

Endpoint: `POST /webhook/bitrix/`

Парсинг `application/x-www-form-urlencoded` от Bitrix:

| Поле | Значение |
|------|---------|
| `data[CONNECTOR]` | `arbitrasupportbot` |
| `data[MESSAGES][0][chat][id]` | `telegram_chat_id` (ключ для поиска сессии) |
| `data[MESSAGES][0][message][text]` | Текст от оператора |
| `data[MESSAGES][0][message][files][N][link]` | URL файла (если есть) |
| `data[MESSAGES][0][im][chat_id]` | Внутренний Bitrix chat_id |

Логика обработки:
1. Проверить `data[CONNECTOR] == "arbitrasupportbot"` → иначе 200 OK и игнор
2. Найти сессию в `bot_sessions` по `chat_id`
3. Если `session.escalated == False` → 200 OK и игнор (запоздавший webhook)
4. Очистить BB-коды (`[b]...[/b]`, `[i]...[/i]`, `[br]`, `[url=...]....[/url]`)
5. Отправить текст клиенту через `bot.send_message(chat_id, cleaned_text)`
6. Если есть файлы — отправить ссылки (файлы как текст в этой версии)
7. Обновить `operator_last_reply_at = now()` в `bot_sessions`

#### Схема расширения `bot_sessions`

Новые колонки в Supabase таблице `bot_sessions`:

| Колонка | Тип | Назначение |
|---------|-----|-----------|
| `escalated` | `bool` default `false` | Клиент сейчас с оператором |
| `bitrix_chat_id` | `text` nullable | Bitrix session CHAT_ID (из ответа ImConnector) |
| `escalated_at` | `timestamptz` nullable | Когда началась эскалация |
| `operator_last_reply_at` | `timestamptz` nullable | Когда оператор последний раз написал |

#### Стейт-машина: маршрутизация сообщений

**Клиент пишет сообщение → `handlers/text.py`:**

```
session.escalated == True?
  ├─ Да → forward_to_bitrix()  (не через ИИ)
  └─ Нет → support_svc.answer()
            └─ switcher=true? → trigger_escalation()
```

`forward_to_bitrix()` — отправляет сообщение в Bitrix через `imconnector.send.messages.json` (тот же метод, но без истории — просто текст очередного сообщения).

#### Логика возобновления ИИ

Три события, при которых `escalated` → `False` и ИИ снова включается:

**1. Таймаут без ответа оператора (1 час)**

Фоновая asyncio-задача `escalation_watchdog()` запускается в `bot.py`:
- Каждые 5 минут опрашивает `bot_sessions` где `escalated=True`
- Если `now() - operator_last_reply_at > 1 час` → `escalated = False` + отправить клиенту:
  > "Специалист завершил консультацию. Если у вас остались вопросы — я готова помочь."
- Если оператор вообще не ответил (проверяем `operator_last_reply_at IS NULL AND now() - escalated_at > 1 час`) → тот же сброс

**2. Оператор закрывает чат в Bitrix**

Bitrix шлёт `POST /webhook/bitrix/` с событием `IMOPENLINES.SESSION.FINISH`:
- Webhook-сервер парсит событие, находит сессию, сбрасывает `escalated = False`
- Клиенту отправляется:
  > "Наш специалист завершил сессию. Как всё прошло? Если остались вопросы — я готова помочь."

**3. Клиент пишет после возобновления**

После сброса `escalated = False` — следующее сообщение клиента обрабатывается ИИ в штатном режиме.

## ⚠️ Риски и ограничения

- **Bitrix поля документов**: 16 основных полей чеклиста подтверждены и вшиты в спецификацию. Значения null у ранних сделок (C4:NEW) — норма: в этом случае профиль показывает "документы ещё не собраны".
- **Разрастание промптов**: инжект `COMPANY_FACTS` + история + контекст может существенно увеличить расход токенов на агентов R1/R2. Необходимо измерить и при необходимости сократить history limit для R1.
- **Тестовый INN**: для `quality_run.py` нужен тестовый ИНН с данными в support Supabase — иначе `client_context` будет пустым и оценка будет нерепрезентативной.
- **OAuth токены**: ✅ извлечены из Redis на VPS и добавлены в `.env`. `refresh_token` проверен — обмен работает. `access_token` истекает через 1 час, автообновление реализуется в T14.
- **Регистрация коннектора**: однократный ручной шаг перед T14 (скрипт-референс: `/home/webuser/Arbitra/testing_calling_bitrix.py`). Выполнять только после того как webhook-сервер будет запущен и доступен из интернета.
- **Webhook URL и сеть**: бот ещё не задеплоен на VPS (Docker пуст). Firewall на VPS: открыты 22, 80, 443 — порт 8080 закрыт. Bitrix требует HTTPS для `PLACEMENT_HANDLER`. Решение: запустить aiohttp на порту 8080 внутри, проксировать через nginx на `443/webhook/bitrix/` (nginx уже обслуживает порт 443). Нужно добавить `location /webhook/bitrix/` в nginx-конфиг до регистрации коннектора.
- **Публичный URL для webhook**: сервер должен быть доступен из интернета. Если на VPS нет HTTPS — Bitrix может отказать в подключении. Нужно проверить firewall + TLS до T15.
- **Watchdog в polling-боте**: aiogram polling и aiohttp web-сервер в одном процессе — стандартная схема, но нужно проверить graceful shutdown обоих корутин.
