## 🎯 Цель

Клиент должен пройти двухшаговую верификацию (ИНН → телефон) перед получением доступа к боту. Все граничные сценарии — повторные ошибки, телефон не совпал, Bitrix недоступен, уже авторизован — должны быть явно обработаны с понятными сообщениями, а не «застревать» в состоянии без выхода.

---

## 📦 Scope

### Входит:

- [ ] Обработка `/start` для уже авторизованного клиента (T01)
- [ ] Счётчик ошибок ИНН с двухэтапной эскалацией (T02)
- [ ] Эскалация при несовпадении телефона (T03)
- [ ] Отдельный ответ при недоступности Bitrix (T04)
- [ ] Консультативный режим во время авторизации (T05)
- [ ] Автотесты через Telegram MCP (T06)

### Не входит:

- Повторная авторизация / смена аккаунта
- 10-значный ИНН (юридические лица) — не поддерживается в MVP
- Меню и поставоризационные сценарии

---

## 🏗 Архитектура и структура

### State machine (полная)

```
/start (любое состояние)
  ├── если AUTHORIZED → "вы уже авторизованы" + открыть меню
  └── иначе → сбросить сессию, WAITING_INN

WAITING_INN
  ├── текст без 12-значного ИНН → AI (no_inn_in_text), error_count++
  │     └── error_count >= 2 → AI + упоминание @Lobster_21
  ├── ИНН найден в Bitrix → WAITING_PHONE, просим поделиться телефоном
  ├── ИНН не найден в Bitrix → AI (inn_not_found), error_count++
  │     └── error_count >= 2 → AI + упоминание @Lobster_21
  └── Bitrix недоступен (исключение) → "технические проблемы, попробуйте позже"

WAITING_PHONE
  ├── F.contact + телефон совпал → AUTHORIZED, приветствие + меню
  ├── F.contact + телефон не совпал → AI (phone_mismatch) + упоминание @Lobster_21
  └── F.text → AI (waiting_for_phone), напоминание про кнопку
        └── если вопрос по процедуре → Алина отвечает + возвращает к шагу
```

### Консультативный режим (во всех auth-состояниях)

Если клиент в любом из состояний `WAITING_INN` / `WAITING_PHONE` задаёт вопрос по процедуре (где найти ИНН, зачем нужен телефон, что такое ИНН и т.д.) — Алина отвечает на вопрос и в конце того же сообщения возвращает клиента к текущему шагу.

Реализация: через системный промпт — GPT получает текущее состояние как контекст и инструкцию "после ответа на вопрос напомни клиенту о текущем шаге". Новое состояние не вводится.

### Правила счётчика `error_count`

- Увеличивается при каждой неудачной попытке в состоянии `WAITING_INN`
  (невалидный ИНН + ИНН не найден в Bitrix — оба типа считаются)
- Этап 1 (error_count = 1): стандартный AI-ответ
- Этап 2 (error_count >= 2): AI-ответ + предложение написать в @Lobster_21
- Сбрасывается при `/start` вместе с остальной сессией
- Не меняется при успешном нахождении ИНН

## 🧪 Acceptance Criteria

- [ ] `/start` при AUTHORIZED не сбрасывает сессию, показывает меню
- [ ] После 1-й ошибки ИНН — стандартный AI-ответ без упоминания поддержки
- [ ] После 2-й ошибки ИНН — в ответе есть упоминание @Lobster_21
- [ ] При несовпадении телефона — в ответе есть упоминание @Lobster_21
- [ ] При недоступности Bitrix — клиент видит "технические проблемы", а не "ИНН не найден"
- [ ] В `WAITING_INN`: вопрос "где найти ИНН?" → Алина объясняет + напоминает про ввод ИНН
- [ ] В `WAITING_PHONE`: вопрос "зачем телефон?" → Алина объясняет + кнопка поделиться
- [ ] Автотесты T06 проходят зелёным по всем сценариям state machine

## ⚠️ Риски

- `error_count` хранится в Supabase-сессии. Если сессия по какой-то причине пересоздалась (RPC `get_or_create_session`), счётчик обнулится — это нормально для MVP.
- Контакт Alina с поддержкой: `@Lobster_21` — захардкожен в промптах. При смене контакта нужно обновить OpenAI-промпты.
