# T08_s01_prompt_tuning — Промпт-тюнинг на реальных делах

## Customer-facing инкремент

Ответы Алевтины проверены на 2–3 реальных делах. Гришин увидит точные, структурированные, человечные ответы без галлюцинаций.

## Цель

Протестировать оба сценария (`case_status` и `next_steps`) на реальных делах с разными стадиями. Итеративно улучшить промпт до качества, готового к демо.

## Контекст

T04–T07 завершены — полный диалоговый флоу работает. Бот задеплоен локально или на тестовой среде. Выбрать 2–3 дела с хорошо заполненными данными в `electronic_case`.

## Scope

**Подготовка:**
- Выбрать 2–3 дела с заполненными полями в `electronic_case` (стадия, суд, документы, финансы)
- Убедиться что `get_case_context(inn)` возвращает содержательный текст по каждому из них
- Авторизоваться за каждого клиента (ИНН + телефон)

**Тест-сценарий для каждого дела:**
1. «Что с моим делом?» → оценить: точность, структура, лаконичность, отсутствие выдуманных фактов
2. «Что мне делать дальше?» → оценить: соответствие стадии, конкретность шагов, не-общие советы
3. «Привет» → приветствие без лишнего
4. «Спасибо» → «Рада помочь!»
5. «Расскажи о погоде» → redirect без вопроса в конце

**Критерии качества ответа:**
- Структура: секции разделены, используются `<b>` заголовки
- Лаконичность: нет воды и канцелярита
- Точность: только факты из `case_context`, не выдуманные
- Человечность: не бюрократический стиль
- Финал: ответ не заканчивается вопросом

**Итерация промпта:**
Если ответ не соответствует критериям — корректировать системный промпт в `alevtina_service.py`:
- Часть 1 (личность, правила) — если нарушаются базовые правила
- Часть 2 (БФЛ-знание) — если шаги неточные или не соответствуют стадии
- Добавить примеры в промпт (few-shot), если GPT стабильно неправильно форматирует

**Типичные проблемы и фиксы:**
- Галлюцинации → усилить правило «только факты из данных дела»
- Ответ заканчивается вопросом → добавить примеры без вопроса
- Слишком длинный ответ → добавить ограничение по словам или структуре
- Стадия не распознана → проверить формат `stage` в `case_context`, уточнить маппинг в промпте

## Out of scope

- Изменение бизнес-логики (интенты, кеш, история) — T04–T07 закрыты
- Изменение DDL или структуры сессий
- Деплой — T09

## Технические детали

Тюнинг не требует изменения кода за пределами строк-констант в `alevtina_service.py` (PART1_PERSONALITY, PART2_BFL). Изменения промпта — это строки в Python-файле, не конфиг.

`temperature=0.3` — стандартное значение. Если ответы слишком однообразны, можно поднять до 0.5. Если слишком нестабильны — опустить до 0.1.

## Как протестировать

### Чек-лист перед завершением T08

- [ ] case_status: ответ точный, без выдуманных фактов — на 2 разных делах
- [ ] next_steps: шаги соответствуют реальной стадии — на 2 разных стадиях
- [ ] greeting: приветствие адекватное, не избыточное
- [ ] gratitude: «Рада помочь!» — дословно или близко
- [ ] off_topic: redirect без вопроса, без предложений сторонних тем
- [ ] Ответы не заканчиваются вопросом — проверено на 5 разных запросах
- [ ] HTML-разметка рендерится корректно в Telegram

## Критерии приёмки

1. Все пункты чек-листа пройдены
2. Промпт зафиксирован в `alevtina_service.py` (не черновик)
3. Подобраны 2–3 конкретных дела (ИНН) для демо — зафиксированы в комментарии или заметке
