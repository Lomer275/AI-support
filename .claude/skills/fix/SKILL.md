# Fix Skill

Планирование и применение фиксов из код-ревью (CRITICAL → HIGH → MEDIUM).
Минимальные точечные изменения. Без подтверждения не начинает.

---

## Входные данные

Пользователь предоставляет одно или несколько из:
- **Таблицу находок** из `/codereview` или `/visualcheck` (вставляет в чат)
- **Файл задачи** с секцией `## Code Review`
- **Конкретный баг** для фикса (описание словами)
- **Код файлов** для исправления

Если нет таблицы находок — попроси сначала запустить `/codereview`.

---

## Алгоритм выполнения

### Шаг 1 — Сбор контекста

1. Найди и прочитай **таблицу находок** (из ревью или от пользователя)
2. Прочитай **исходный код** затронутых файлов (пользователь должен предоставить)
3. Отфильтруй баги по severity: CRITICAL → HIGH → MEDIUM, пропусти уже помеченные FIXED
4. Если баги UI/* из visualcheck — уточни какие из них нужно фиксить

---

### Шаг 2 — Составление плана (ОБЯЗАТЕЛЬНО перед любыми изменениями)

Для каждого бага составь план:

```
## План фиксов

### R1 — CRITICAL — [Краткое название]
**Root cause:** Почему это происходит (конкретно)
**Файл:** path/to/file.ts
**Изменение:** Что именно меняем (строки N–M)
**Минимальный фикс:** [одна строка или минимальный блок]
**Риск:** Низкий / Средний / Высокий — почему

### R2 — HIGH — [Краткое название]
...

**Итого:** N фиксов, затронуто M файлов
```

После плана — **остановись и жди подтверждения** пользователя.

Фразы-триггеры для продолжения: "делай", "go", "применяй", "ок", "давай", "+".

---

### Шаг 3 — Выполнение фиксов

После подтверждения:

1. **Фикси по одному багу** — один баг = одно минимальное изменение
2. **Не рефакторь попутно** — только то что в плане
3. **Показывай изменения** в формате diff или "было / стало":

```
### Фикс R1 — [Название]

**Было:**
```ts
// проблемный код
const result = query + userInput; // SQL injection
```

**Стало:**
```ts
// фикс
const result = db.query('SELECT * WHERE id = ?', [userInput]);
```
```

4. После каждого файла — сообщи что изменено

---

### Шаг 4 — Верификация

После применения всех фиксов:

**Попроси пользователя** проверить:
- Запустить приложение / тесты
- Пройти сценарий который воспроизводил баг
- Для UI фиксов — сделать скриншот и загрузить

**Если пользователь вернул ошибку** — анализируй, предлагай скорректированный фикс. До 5 итераций. Если не удалось — честно пишешь `❌ НЕ УДАЛОСЬ починить за 5 попыток` с описанием что пробовал.

---

### Шаг 5 — Сводка

```
## Результат фиксов

| ID | Severity | Статус | Комментарий |
|----|----------|--------|-------------|
| R1 | CRITICAL | ✅ FIXED | Параметризованный запрос в api/users.ts:34 |
| R2 | HIGH | ✅ FIXED | clearToken() добавлен в logout() |
| R3 | MEDIUM | ⚠️ PARTIAL | Обработан undefined, но не null — нужен доп. фикс |
| R4 | LOW | ⏭️ SKIP | Пропущен по договорённости |

**Итого:** 2 FIXED, 1 PARTIAL, 1 SKIP
**Рекомендация:** Запустить /codereview повторно для верификации
```

---

## Правила

- **Без подтверждения не начинать** — план сначала, потом ждать "делай"
- **Минимальный фикс** — не рефакторить, не переписывать, не улучшать попутно
- **Один баг = одно изменение** — не смешивать несколько фиксов в один блок
- **Если файл не предоставлен** — попроси его перед тем как предлагать фикс
- **Порядок**: CRITICAL → HIGH → MEDIUM, LOW — по договорённости с пользователем
- **Не угадывай контекст** — если непонятно как устроен модуль, попроси показать связанный код
- **Честность по таймауту**: если за 5 итераций не удалось — признай, не изобретай workaround без предупреждения

---

## Быстрый режим

Если пользователь говорит "фикси всё без остановки" / "go straight" — можно пропустить ожидание подтверждения и сразу применять план. Но план всё равно показать перед фиксами.
