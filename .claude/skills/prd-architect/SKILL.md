# PRD-Architect Skill

Создаёт и развивает документацию проекта по стандартам SNN/BR. Только документирование — без написания кода.

---

## Входные данные

Пользователь предоставляет одно или несколько из:
- Описание фичи/идеи в свободной форме
- Ссылку на бизнес-требования (BR)
- Тип документа: спецификация (SNN), бизнес-требования (BR), архитектурный документ
- Контекст: что уже есть, что меняется

Если не предоставлено ничего — задай уточняющие вопросы (одним сообщением).

---

## Алгоритм выполнения

### Шаг 0 — Определить тип документа

**Спецификация (SNN):** техническое описание инкремента для разработки. Создаётся, когда нужно описать **что именно строим и как**.

**Бизнес-требования (BR):** описание проблемы и пожеланий бизнеса. Создаётся **до** спецификации. Отвечает на вопрос **зачем**.

**Архитектурный документ:** описание технических решений, компонентов, интеграций.

Если тип не ясен — уточни у пользователя.

---

### Шаг 1 — Собрать контекст

Прочитай существующие документы для понимания контекста:

1. `SUP-HANDOFF.md` — текущий статус проекта, активные задачи
2. `SUP-architecture.md` — архитектура системы
3. Если пишется новая спека — прочитай смежные спеки из `docs/2. SUP-specifications/`
4. Если есть BR — прочитай соответствующий файл из `docs/1. SUP-business requirements/`

Прочитай гайды (только при первом создании документа данного типа или если есть сомнения):
- `docs/4. SUP-guides/specifications_guide.md` — для SNN
- `docs/4. SUP-guides/business_requirements_template.md` — для BR
- `docs/4. SUP-guides/doc_conventions.md` — правила именования

---

### Шаг 2 — Определить нумерацию

**Для новой спеки (SNN):**
- Проверь существующие файлы: `glob docs/2. SUP-specifications/S*.md`
- Найди максимальный SNN → следующий = max + 1
- Если файлов нет — начинай с S01

**Для нового BR:**
- Проверь: `glob docs/1. SUP-business requirements/*.md`
- Определи логический номер из существующей нумерации

---

### Шаг 3 — Задать уточняющие вопросы (если данных недостаточно)

Прежде чем писать документ, убедись что понимаешь:

```
Прежде чем создать спецификацию, уточню несколько деталей:

1. **Цель / Проблема** — какую проблему решаем? Что не работает или чего не хватает?
2. **Пользователь** — кто получает ценность? (клиент, оператор, система)
3. **Ключевые сценарии** — опиши 2-3 главных use case
4. **Ограничения** — что точно НЕ входит в scope?
5. **Зависимости** — есть ли зависимости от других спек или систем?
6. **Приоритет** — когда это нужно? Что блокирует?
```

Жди ответа перед созданием документа.

## Работа с существующими документами

### Обновление спеки

Если пользователь просит обновить существующую спеку:
1. Прочитай текущий файл
2. Определи что меняется (добавляется фаза, обновляется статус, добавляются задачи)
3. Обнови версию (1.0 → 1.1) и дату
4. Добавь секцию истории изменений в конец файла:

```markdown
## 📝 История изменений

### v1.1 (YYYY-MM-DD)
- <Что изменилось>

### v1.0 (YYYY-MM-DD)
- Первая версия
```

### Ревью спеки

Если пользователь просит проверить спеку — проверь по чек-листу:

- [ ] Заполнен заголовок (ID, статус, версия, дата)
- [ ] Описана цель (1-2 абзаца, какую проблему решаем)
- [ ] Чётко определён scope (что входит / что не входит)
- [ ] Описана архитектура (какие модули затрагиваются)
- [ ] Описаны модели данных / изменения БД (если есть)
- [ ] Перечислены задачи (TNN) с зависимостями и статусами
- [ ] Написаны acceptance criteria (DoD) для каждой фазы
- [ ] Добавлены ссылки на связанные документы

---

## Правила

- **Только документирование** — не пиши код, не создавай задачи (это делает `/task-planner`)
- Инженерный, сухой стиль — без лирики, без «возможно», без воды
- Один спринт (SNN) = один функциональный инкремент
- Никогда не перезаписывай существующие документы без явного подтверждения пользователя
- Дата в документах — всегда сегодняшняя (`currentDate` из системного контекста)
- Если не хватает контекста — спроси, не додумывай
