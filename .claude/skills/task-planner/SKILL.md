# Task-Planner Skill

Декомпозирует спецификации на задачи и создаёт файлы задач по стандарту проекта.

---

## Входные данные

Пользователь предоставляет одно из:
- Номер спеки (`S05`, `S06`) или путь к файлу
- Описание функционала для декомпозиции
- Просьбу создать конкретную задачу

Если не предоставлено ничего — попроси указать спеку или описать что нужно реализовать.

---

## Алгоритм выполнения

### Шаг 1 — Изучить спецификацию

1. Прочитай файл спеки из `docs/2. SUP-specifications/SNN_*.md`
2. Прочитай `SUP-HANDOFF.md` — понять текущий контекст, активные задачи, приоритеты
3. Если есть связанные BR — прочитай из `docs/1. SUP-business requirements/`
4. Прочитай гайд (при первом использовании или при сомнениях): `docs/4. SUP-guides/task_decomposition_guide.md`

---

### Шаг 2 — Определить следующий TNN

Найди максимальный номер задачи по всем файлам:
```
glob docs/3. SUP-tasks/**/*.md
```

Найди максимальное число в именах файлов вида `TNN_*.md`.
Следующий номер = max + 1.

Если задач нет — начинай с T01.

---

### Шаг 3 — Составить план декомпозиции

Прежде чем создавать файлы — предложи пользователю план:

```
## План декомпозиции SNN: <Название>

Предлагаю разбить на N задач:

### Порядок выполнения:

**T[N] — <Название>**
- Инкремент: <что получает пользователь>
- Зависит от: —
- Блокирует: T[N+1]
- Размер: ~<маленький/средний/большой>

**T[N+1] — <Название>**
- Инкремент: <что получает пользователь>
- Зависит от: T[N]
- Параллельно с: T[N+2] (если возможно)
- Размер: ~<маленький/средний/большой>

---
Принципы декомпозиции:
- По инкрементам, не по слоям
- Каждая задача = проверяемый результат
- Зависимости явные

Утверждаем этот план или нужно что-то изменить?
```

**Жди подтверждения перед созданием файлов.**

---

### Шаг 4 — Создать файлы задач

После подтверждения создай каждую задачу.

#### Структура папок для задач

```
docs/3. SUP-tasks/
└── SNN_<spec_name>/
    ├── TNN_sNN_<task_name>.md
    └── TNN+1_sNN_<task_name>.md
```

Папка называется `SNN_<spec_name>` (без слова "spec").

#### Шаблон файла задачи

Файл: `docs/3. SUP-tasks/SNN_<spec_name>/TNN_sNN_<task_name>.md`

```markdown
Дата: YYYY-MM-DD
Статус: draft
Спецификация: docs/2. SUP-specifications/SNN_<spec_name>.md

# TNN_sNN_<task_name> — <Краткое описание>

## Customer-facing инкремент

<Пользователь может ... и получает ...>

## Цель

<Что меняется в поведении системы>

## Контекст

<Текущее состояние, почему нужна эта задача>

## Scope

- <Конкретный перечень изменений>
- <Какие файлы/модули затрагиваются>

## Out of scope

- <Что точно не делаем в этой задаче>

## Технические детали

## Как протестировать

### 1. Базовый сценарий

1. <Шаг 1>
2. <Шаг 2>
3. Проверить:
   - <что A работает>
   - <что B сохраняется>

### 2. Негативный сценарий

1. <Что должно происходить при ошибке>

## Критерии приёмки

1. <Конкретный проверяемый факт>

3. <Конкретный проверяемый факт>

## Правила завершения задачи

**После акцепта задачи AI выполняет автоматически:**

1. **Обновить файл задачи:**
   - Статус → `✅ accepted`
   - Добавить дату завершения

2. **Переместить файл:**
   - Из: `docs/3. SUP-tasks/SNN_*/TNN_*.md`
   - В: `docs/3. SUP-tasks/Done/SNN_*_done/TNN_*_done.md`

3. **Обновить спецификацию:**
   - Ссылки на задачи → новые пути
   - Статусы → обновить

4. **Обновить CHANGELOG.md:**
   - Добавить запись в секцию `[Не выпущено] → Добавлено`

5. **Обновить HANDOFF.md:**
   - Отметить задачу как выполненную
   - Обновить связанные секции
```

---

### Шаг 5 — Обновить спецификацию

Добавь или обнови секцию `📋 Задачи` в спеке:

```markdown
## 📋 Задачи

| ID | Задача | Зависит от | Статус |
|----|--------|------------|--------|
| T[N] | [<Название>](../3. SUP-tasks/SNN_.../TNN_sNN_<name>.md) | — | 🔵 planned |
| T[N+1] | [<Название>](../3. SUP-tasks/SNN_.../TNN+1_sNN_<name>.md) | T[N] | 🔵 planned |
```

---

### Шаг 6 — Обновить HANDOFF.md

Спроси пользователя про приоритет задач, затем добавь в соответствующий раздел:

```
Какой приоритет для задач SNN?
- P1 (в работу сейчас)
- P2 (следующий спринт)
- Backlog (отложить)
```

**После ответа** добавь в `SUP-HANDOFF.md`:

```markdown
### <SNN>: <Название спека>

**Порядок выполнения:**

| ID | Задача | Зависит от | Статус |
|----|--------|------------|--------|
| [T[N]](docs/3.%20SUP-tasks/...) | <Название> | — | 🔵 planned |
| [T[N+1]](docs/3.%20SUP-tasks/...) | <Название> | T[N] | 🔵 planned |
```

```

Создано задач: N
  T[N] — <Название> (docs/3. SUP-tasks/...)
  T[N+1] — <Название> (docs/3. SUP-tasks/...)

Следующий шаг: разработка T[N] или /orchestrate для параллельного запуска
```

## Принципы декомпозиции (краткая выжимка)

**По инкрементам, не по слоям:**
- ❌ Плохо: T01 — модели БД, T02 — handlers, T03 — тесты
- ✅ Хорошо: T01 — пользователь может авторизоваться, T02 — пользователь видит меню

**Размер задачи:** не более 50% контекста GPT — примерно 1-3 файла handlers/routers, 1-2 модели данных или API endpoint.

**Зависимости:**
- Блокирует: A должна завершиться до начала B
- Параллельно с: A и B независимы, могут выполняться одновременно

**Технические задачи (рефакторинг, миграции) требуют явного обоснования "зачем".**

---

## Правила

- Всегда жди подтверждения плана перед созданием файлов
- Создание файлов задач БЕЗ добавления в HANDOFF.md — незавершённая работа
- Дата в файлах — всегда сегодняшняя (`currentDate` из системного контекста)
- Не создавай задачи больше чем нужно — только то что описано в scope спеки
- Не пиши код в задачах, только описание
